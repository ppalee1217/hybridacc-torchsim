1. 問題摘要

🐛 Bug 現象

  • Gate GEMM single-wave：每 48 個 fp16 輸出中，前 12 個與 RTL/golden 吻合，後 36 個以週期 3 重複前段。
  • Root cause 鎖定在 p2pe0.exe_a_stage 的 S_WAIT_PLO release cycle。
  • 18/18 個 S_WAIT_PLO settled PLO sample 滿足 plo == vaddu_result_reg， 且 plo != vaddu_result_sig（stale replay）。
  • RTL pre-sim 不重現：RTL 踩到同一條路徑，但 pop+produce 同拍的 rollover 語義在 RTL 正確，gate 失真。

✅ 修正方向

  • 保留現有 8-state FSM encoding，不新增 state，不動 PErouter / ProcessElement。
  • 新增 1-bit plo_pending_valid_reg：顯式記錄「有待補回 buffer 的 pending VPSUM」。
  • 把 S_WAIT_PLO 的 release 拆成兩個 cycle：先 drain（僅 pop），再 refill（僅 produce）。
  • 消除 plo_buf_do_pop_w && plo_buf_produce_w 同拍成立的可能性。
  • S_WAIT_PLO 期間 ready_out_w 恆為 0，禁止 upstream 同拍前推。

2. 訊號相依圖（Bug 根源）

下圖顯示 bug 的組合邏輯相依關係：plo_ready 同時影響 plo_buf_do_pop_w 與 plo_buf_can_push_w，導致在 S_WAIT_PLO 的 release cycle 中，pop 與 produce 同拍成立。

圖 2-1：修改前的組合邏輯相依圖。plo_ready 同時驅動 pop 與 produce，在 S_WAIT_PLO 產生同拍衝突。
關鍵觀察： plo_buf_can_push_w = !plo_buf_valid_reg || plo_ready。 在 S_WAIT_PLO 的 release cycle（plo_buf_valid_reg=1, plo_ready=1）， can_push_w=1 且 do_pop_w=1 同時成立， 導致 plo_buf_produce_w=1 與 plo_buf_do_pop_w=1 在同一拍並行觸發。 RTL 在此情況下能維持正確 rollover，但 gate 失真成 stale replay。

3.1 Bug 時序（修改前）

圖 3-1：修改前時序。T₂ 為 S_WAIT_PLO release cycle，do_pop_w 與 produce_w 同拍成立，buf_data_reg 被 stale value 覆寫。

3.2 Fix 時序（修改後）

圖 3-2：修改後時序。T₂ 僅執行 drain（pop），T₃ 才執行 refill（produce），兩者不再重疊。buf_data_reg 在 T₄ 後含正確的 vpsum_NEW。

以下重點放在 PLO 相關的三個狀態。

5. RTL 修改明細

① 新增 plo_pending_valid_reg / _next 暫存器

在 pipeline register 宣告區與 next-state signal 區各加一個 1-bit 控制旗號，記錄「有待補回 PLO buffer 的 pending result」。

✖ BEFORE — EXE_A_Stage.sv (L65–66, L81–82)
// PLO output buffer
logic             plo_buf_valid_reg;
logic [63:0]      plo_buf_data_reg;

// next-state signals
logic             plo_buf_valid_next;
logic [63:0]      plo_buf_data_next;
✔ AFTER — 新增 pending_valid 控制位元
// PLO output buffer
logic             plo_buf_valid_reg;
logic [63:0]      plo_buf_data_reg;
logic             plo_pending_valid_reg;  // ← NEW

// next-state signals
logic             plo_buf_valid_next;
logic [63:0]      plo_buf_data_next;
logic             plo_pending_valid_next; // ← NEW

② plo_buf_produce_w — 拆成互斥的 execute produce 與 refill produce

原有邏輯把 S_EXEC_PLI_VADDU 與 S_WAIT_PLO 合併在同一個 produce 條件， 導致 S_WAIT_PLO 的 release cycle 可以與 pop 同拍觸發 produce。 修改後，兩個路徑條件互斥，S_WAIT_PLO 的 produce 只在 buffer 已清空且 pending_valid=1 時才觸發。

✖ BEFORE — L259–265
always_comb begin
    plo_buf_do_pop_w   = plo_buf_valid_reg && plo_ready;
    plo_buf_can_push_w = !plo_buf_valid_reg || plo_ready;
    plo_buf_produce_w  = (state_reg == S_EXEC_PLI_VADDU
                        || state_reg == S_WAIT_PLO)
                       && plo_buf_can_push_w;
    plo_buf_vpsum_data_w = (state_reg == S_WAIT_PLO)
        ? v_fp16_to_u64(vaddu_result_reg)
        : v_fp16_to_u64(vaddu_result_sig);
end
✔ AFTER — produce 條件拆開，pop 與 refill 不再同拍
always_comb begin
    plo_buf_do_pop_w   = plo_buf_valid_reg && plo_ready;
    plo_buf_can_push_w = !plo_buf_valid_reg || plo_ready;
    // execute produce (S_EXEC_PLI_VADDU): 使用 vaddu_result_sig，行為不變
    // refill produce (S_WAIT_PLO):      只在 buf 已空且 pending=1 時觸發，
    //                                   絕不與 do_pop_w 同拍成立
    plo_buf_produce_w  =
        (state_reg == S_EXEC_PLI_VADDU && plo_buf_can_push_w) ||
        (state_reg == S_WAIT_PLO
         && !plo_buf_valid_reg           // buffer 已清空
         && plo_pending_valid_reg);       // 有待補的 result
    plo_buf_vpsum_data_w = (state_reg == S_WAIT_PLO)
        ? v_fp16_to_u64(vaddu_result_reg)
        : v_fp16_to_u64(vaddu_result_sig);
end
為什麼 S_EXEC_PLI_VADDU 的同拍 pop+produce 可以保留？
在 S_EXEC_PLI_VADDU，produce 使用的是 vaddu_result_sig（當前計算結果）， 不是 vaddu_result_reg（舊 cycle 的 stale 值）。 這個 rollover 在 RTL 與 gate 都是安全的，因為寫入的是新鮮計算值，不是舊值再覆寫。 只有 S_WAIT_PLO 的 refill 是問題所在。

③ PLO Buffer Next-State always_comb — 加入 plo_pending_valid_next 管理

✖ BEFORE — L269–287
always_comb begin
    plo_buf_valid_next = plo_buf_valid_reg;
    plo_buf_data_next  = plo_buf_data_reg;

    if (stage_reset) begin
        plo_buf_valid_next = 1'b0;
        plo_buf_data_next  = 64'd0;
    end else begin
        if (plo_buf_do_pop_w && !plo_buf_produce_w) begin
            plo_buf_valid_next = 1'b0;
        end else if (!plo_buf_do_pop_w && plo_buf_produce_w) begin
            plo_buf_valid_next = 1'b1;
            plo_buf_data_next  = plo_buf_vpsum_data_w;
        end else if (plo_buf_do_pop_w && plo_buf_produce_w) begin
            plo_buf_valid_next = 1'b1;
            plo_buf_data_next  = plo_buf_vpsum_data_w;
        end
    end
end
✔ AFTER — 加入 pending_valid 清除（refill 後歸零）
always_comb begin
    plo_buf_valid_next    = plo_buf_valid_reg;
    plo_buf_data_next     = plo_buf_data_reg;

    if (stage_reset) begin
        plo_buf_valid_next = 1'b0;
        plo_buf_data_next  = 64'd0;
    end else begin
        if (plo_buf_do_pop_w && !plo_buf_produce_w) begin
            plo_buf_valid_next = 1'b0;
        end else if (!plo_buf_do_pop_w && plo_buf_produce_w) begin
            plo_buf_valid_next = 1'b1;
            plo_buf_data_next  = plo_buf_vpsum_data_w;
        end else if (plo_buf_do_pop_w && plo_buf_produce_w) begin
            // 修改後此 branch 只出現在 S_EXEC_PLI_VADDU
            plo_buf_valid_next = 1'b1;
            plo_buf_data_next  = plo_buf_vpsum_data_w;
        end
    end
end
注意： plo_pending_valid_next 的主要管理（set / clear）移到 Main FSM always_comb 中， 與 state_next 的轉換一起維護，避免跨 always_comb block 多重驅動。

④ ready_out_w — S_WAIT_PLO 改為恆 0

修改前：ready_out_w = plo_ready，導致 upstream 可在 release 同拍接受新指令。 修改後：等待期間禁止前推，確保 drain/refill 兩拍不被新指令污染。

✖ BEFORE — L171–172
S_WAIT_PLO: begin
    ready_out_w = plo_ready;
end
✔ AFTER — 等待期間禁止 upstream 前推
S_WAIT_PLO: begin
    ready_out_w = 1'b0;  // drain/refill 期間不接受新指令
end

⑤ pli_ready — 移除 S_WAIT_PLO 的 PLI accept 條件

修改前 S_WAIT_PLO 允許接受新的 PLI，在等待期間可能導致 pli_data_reg 被覆寫。

✖ BEFORE — L303–307
end else if (state_reg == S_IDLE
          || state_reg == S_EXEC_PLI_VADDU
          || state_reg == S_WAIT_PLO) begin
    pli_ready = accept_vpsum;
end
✔ AFTER — S_WAIT_PLO 期間拒絕新 PLI
end else if (state_reg == S_IDLE
          || state_reg == S_EXEC_PLI_VADDU) begin
    pli_ready = accept_vpsum;
end
// S_WAIT_PLO: pli_ready 保持預設值 0

⑥ Main FSM — S_EXEC_PLI_VADDU：進入 S_WAIT_PLO 時置 pending_valid=1

✖ BEFORE — S_EXEC_PLI_VADDU (L472–475)
S_EXEC_PLI_VADDU: begin
    vaddu_result_next = vaddu_result_sig;

    if (!(!plo_buf_valid_reg || plo_ready)) begin
        // PLO buffer full → wait
        state_next = S_WAIT_PLO;
    end else begin
        // ... normal advance ...
    end
end
✔ AFTER — 同時置 plo_pending_valid_next=1
S_EXEC_PLI_VADDU: begin
    vaddu_result_next = vaddu_result_sig;

    if (!(!plo_buf_valid_reg || plo_ready)) begin
        // PLO buffer full → wait，記錄 pending result
        state_next             = S_WAIT_PLO;
        plo_pending_valid_next = 1'b1;  // ← NEW
    end else begin
        // ... normal advance ...
    end
end

⑦ Main FSM — S_WAIT_PLO：改成兩相控制

最關鍵的 FSM 變更：修改前用 plo_ready 決定離開時機；修改後改用 plo_buf_valid_reg 與 plo_pending_valid_reg 做兩階段判斷，讓 drain 與 refill 在不同 cycle 完成。

✖ BEFORE — S_WAIT_PLO (L508–540)
S_WAIT_PLO: begin
    if (!plo_ready) begin
        state_next = S_WAIT_PLO;
    end else begin
        // plo_ready=1: release → 同拍接受新指令
        if (ready_out_w && valid_in) begin
            decode_next    = EXE_M_decode_signals_in;
            vmul_data_next = vmul_out_in;
            // ... halt / pli_plo / vaddu dispatch ...
        end else begin
            state_next = S_IDLE;
        end
    end
end
✔ AFTER — 兩相：drain（buf_valid=1）→ refill（pending=1）→ 離開
S_WAIT_PLO: begin
    if (plo_buf_valid_reg) begin
        // Phase 1 — buffer 仍有舊 beat，等 pop 完成
        // pop 由 always_comb plo_buf_do_pop_w 在背景執行
        state_next = S_WAIT_PLO;
    end else if (plo_pending_valid_reg) begin
        // Phase 2 — buffer 已空，refill produce 本 cycle 觸發，
        // plo_pending_valid_next = 0 在此清除
        plo_pending_valid_next = 1'b0;
        state_next             = S_IDLE;
    end else begin
        state_next = S_IDLE;
    end
end
兩相控制的生命週期：
  • Phase 1（buf_valid=1）：plo_buf_do_pop_w 由 always_comb 自動觸發，FSM 只需保持 S_WAIT_PLO。
  • Phase 2（buf_valid=0, pending=1）：plo_buf_produce_w（refill produce）由 change② 的條件觸發，FSM 清除 pending 並離開。
  • 兩相之間，ready_out_w=0 保證 upstream 不會插入新指令（change④）。

6. 驗證要點

修改完成後，建議按下列順序驗證：

RTL Invariant（可加入 SVA assertion 或 wave 目視確認）

建議 SVA Assertions
// 1. S_WAIT_PLO 期間不可同拍 pop + produce
assert property (@(posedge clk) disable iff (!reset_n)
    (state_reg == S_WAIT_PLO) |->
    !(plo_buf_do_pop_w && plo_buf_produce_w))
else $error("WAIT_PLO: pop and produce fired simultaneously!");

// 2. S_WAIT_PLO refill 前 buffer 必須為空
assert property (@(posedge clk) disable iff (!reset_n)
    (state_reg == S_WAIT_PLO && plo_buf_produce_w) |->
    !plo_buf_valid_reg)
else $error("WAIT_PLO refill: buffer not empty!");

// 3. S_WAIT_PLO 期間 ready_out 恆 0
assert property (@(posedge clk) disable iff (!reset_n)
    (state_reg == S_WAIT_PLO) |-> (ready_out == 1'b0))
else $error("WAIT_PLO: ready_out should be 0!");

// 4. plo_pending_valid_reg 在離開 S_WAIT_PLO 後必須為 0
assert property (@(posedge clk) disable iff (!reset_n)
    (state_reg == S_WAIT_PLO &&
     $past(state_reg) != S_WAIT_PLO) |->
    (plo_pending_valid_reg == 1'b1))
else $error("Entered WAIT_PLO without pending set!");

回歸測試順序

步驟執行指令通過條件
1. RTL Compile make rtl_compile 無 elaboration error / warning
2. RTL Regress make rtl_regress_gemm_single_wave compare PASS，1536 fp16 吻合數 0
3. Focused RTL Wave make rtl_regress_gemm_single_wave WAVE_DUMP=1 WAVE_DEPTH=9 p2pe0 S_WAIT_PLO waveform 顯示兩相（drain → refill），無同拍 pop+produce
4. Gate Resynthesis make syn_pe 無 latch / 多重驅動警告
5. Gate Regress make gate_regress_gemm_single_wave CLOCK_PERIOD_NS=2.00 compare PASS，12-good / 36-repeat pattern 消失
效能影響說明： 這次修改在每次 PLO backpressure 事件中多增加 1 個 cycle（drain 與 refill 各一拍， 原先合併為一拍）。GEMM workload 中 backpressure 次數有限，對整體 throughput 影響可忽略。 如需恢復 zero-bubble，可在 bug 修正確認後做第二階段優化，不影響本次正確性目標。

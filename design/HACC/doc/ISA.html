HybridAcc Core Controller 規格（HACC Native Core-ISA）

目標：在 不使用 RISC-V CPU core 的前提下，定義可直接驅動 HybridAcc 的精簡 Core-ISA。
核心方向：少數控制指令 + 硬體微序列執行 + descriptor 資料驅動。

快速摘要

控制面定位

  • Core 僅負責提交、同步、錯誤處理
  • wave 細節由 DFU/WSU/PAU/DRU 等硬體單元展開

可擴展策略

  • code O(1), data O(Nwave)
  • 支援數千至數萬 wave 以 descriptor 驅動

程式模型

  • 固定 core stub：HCFG→HLAUNCH→HKICK→HWAIT
  • Conv/GEMM 差異主要落在資料 section

0. 設計目標與邊界

0.1 目標

  1. 定義可獨立實作的 HACC Native ISA（非 RV32I 相容）。
  2. Core 僅處理任務提交與例外；資料搬移與 cluster 控制由硬體完成。
  3. 支援大量 wave 執行且 core 程式維持常數大小。

0.2 邊界

  1. 不變更 PE ALU 指令語意。
  2. 不變更 HDDU/AGU/Cluster/DMA 的既有 MMIO map。
  3. MVP 採單 context、單 command queue。

0.3 原則

  1. ISA 語意停留於工作提交/同步/查狀態層級。
  2. wave prefetch/apply/start/sync/drain 全由硬體狀態機處理。
  3. busy/error/wave_id 等可觀測狀態需可由 CSR 讀取。

1. 架構總覽

1.1 總體架構

+--------------------+        +------------------------------+
| HACC Core Frontend | -----> | HACC Command Queue           |
| (Native ISA)       |        +------------------------------+
|  - PC/Decoder      |                    |
|  - 16 GPR          |                    v
+--------------------+        +------------------------------+
															| HACC Execution Complex        |
															|  - Descriptor Fetch Unit      |
															|  - Wave Scheduler             |
															|  - Profile Apply Unit         |
															|  - Cluster Broadcast Unit     |
															|  - DMA Range Unit             |
															|  - Sync/Timeout Unit          |
															|  - Error/IRQ Unit             |
															+------------------------------+
																						|
																+-----------+-----------+
																| MMIO Bus / Desc Memory |
																+------------------------+

1.2 執行模型

  1. Core Frontend 解碼後送高階命令至 Command Queue。
  2. Execution Complex 依 JobDesc/WaveDesc 自主執行。
  3. Core 可 blocking wait 或 polling 讀狀態。

1.3 粒度

  • Job：table base / policy / wave range
  • Batch：一次提交多個 wave
  • Wave：硬體 scheduler 自動展開

2. 位址模型與 MMIO 版圖

2.1 固定位址

  • CLUSTER_MMIO_BASE = 0x0000_0000
  • CLUSTER_STRIDE = 0x0001_0000
  • DMA_BASE = 0x8000_0000
  • HACC_CSR_BASE = 0x9000_0000
  • DESC_MEM_BASE = 0xA000_0000

2.2 Cluster 位址換算

cluster_addr = CLUSTER_MMIO_BASE + cluster_id * CLUSTER_STRIDE + local_offset

local window：

  1. 0x0000~0x00FF：SPM config
  2. 0x1000~0x1FFF：HDDU passthrough
  3. 0x2000~0x20FF：NoC command

3. HACC 指令（詳細 ISA 規格）

本章定義「非 RISC-V」原生 HACC 指令格式與語意。

3.1 架構狀態

  1. PC：32-bit 程式計數器
  2. GPR[0..15]：16 個 32-bit 通用暫存器，r0=0
  3. PRED：條件旗標（Z/E）
  4. SYS：系統狀態（halt/exception mask）

3.2 指令長度與對齊

  1. 固定 32-bit 指令寬度
  2. 位址需 4-byte 對齊
  3. Little-endian 儲存

3.3 編碼格式

Format-R（雙來源 + 目的）

[31:26] OPC   [25:22] RD   [21:18] RS1   [17:14] RS2   [13:0] FUNC/IMM14

Format-I（來源 + 立即數）

[31:26] OPC   [25:22] RD   [21:18] RS1   [17:0] IMM18

Format-J（控制流）

[31:26] OPC   [25:0] IMM26 (signed, <<2)

3.4 Opcode 類別

  • 0x00~0x0B：Core 控制（MOVI/MOV/ADD/AND/OR/SLL/SRL/CMP/JMP/BR）
  • 0x0C~0x0F：CSR（CSR_RD/CSR_WR）
  • 0x10~0x1F：HACC 控制（CFG/LAUNCH/KICK/WAIT/POLL/ERR/ABORT）
  • 0x20~0x2F：除錯與追蹤（STEP/TRACE/BREAK）

3.5 MVP 指令表

A) 基礎控制

#指令語意
1MOVI rd, imm18rd = sign_ext(imm18)
2MOV rd, rs1rd = rs1
3ADD rd, rs1, rs2rd = rs1 + rs2
4AND rd, rs1, rs2rd = rs1 & rs2
5OR rd, rs1, rs2rd = rs1 | rs2
6SLL rd, rs1, imm5rd = rs1 << imm5
7SRL rd, rs1, imm5rd = rs1 >> imm5
8CMP rs1, rs2設定 PRED.Z = (rs1 == rs2)
9JMP imm26PC = PC + (sign_ext(imm26) << 2)
10BRZ imm26若 PRED.Z==1 則分支

說明：上述 10 條用於最小控制流程，非資料面主力運算 ISA。

B) CSR 存取

  • CSR_WR csr_id, rs1：將 rs1 寫入 CSR
  • CSR_RD rd, csr_id：CSR 讀入 rd

C) HACC 任務控制

  1. HCFG rs1：rs1 指向 JobConfig base（等效寫入 HACC_JOB_BASE_*）
  2. HLAUNCH rs1, rs2：wave_begin=rs1、wave_count=rs2，僅入 queue（non-blocking）
  3. HKICK：觸發 scheduler doorbell
  4. HWAIT rs1：wait mask（bit0 done / bit1 error / bit2 timeout），blocking 等待任一成立
  5. HPOLL rd：rd = HACC_STATUS
  6. HERR rd：rd = ERR_CODE[15:0] | (ERR_AUX[31:16] << 16)
  7. HABORT：中止並清空 inflight
  8. HBREAK：進入 debug halt

3.6 指令時序語意

  1. HCFG/HLAUNCH/HKICK 為順序一致（提交順序 = 可觀察執行順序）。
  2. HWAIT 為強同步點；返回時 STATUS/ERR/LAST_WAVE_ID 必須可見。
  3. HABORT 完成條件：Execution Complex idle + MMIO sequencer drained。

3.7 例外與錯誤碼

ERR_CODE含義
0x01Descriptor bounds error
0x02DMA channel error
0x03Cluster MMIO timeout
0x04Illegal sync policy
0x05Queue overflow
0x06Unsupported capability

ERR_AUX：[7:0]=channel_id、[15:8]=cluster_id、[31:16]=wave_id

4. .hacc 可讀格式

4.1 設計重點

.hacc 以「描述資料 + 執行策略」為主，避免手寫 per-wave 指令碼。

4.2 檔案骨架

program <name> {
	meta { ... }
	hardware { ... }
	tensors { ... }
	pe_programs { ... }
	scan_chain { ... }
	hddu_profiles { ... }
	dma_plan { ... }
	wave_graph { ... }
	runtime_policy { ... }
}

4.3 必填欄位

cluster_countdeploy_maskcluster_stridedma_basewave_countscheduler_policytimeout_cycle

4.4 wave_graph 語意

  • 每筆 wave 需包含：cluster_mask、profile_id、pe_program_id、prefetch_begin/count、drain_begin/count、sync_policy
  • 壓縮語法：repeat(n)、tile_loop(i,j,k)、delta_rule(field, base, step)

4.5 runtime_policy

  • error_action = trap | skip | abort
  • retry_max
  • profile_mode = full | delta | auto
  • completion_bitmap = on | off

5. Compiler/Packager 流程

.hacc
	-> Parser(AST)
	-> Semantic Check
	-> Wave DAG & Policy IR
	-> Descriptor Lowering
			 - JOB_CONFIG
			 - WAVE_DESC
			 - PROFILE_TABLE
			 - DMA_DESC
	-> Core Stub Emit (HACC Native ISA)
	-> Packager (.hap / .elf)

5.2 Semantic Check（必做）

  1. deploy_mask 不超出 cluster_count
  2. wave DAG 無循環
  3. DMA 覆蓋 tensor byte range
  4. profile/pe_program 引用合法
  5. sync policy 與 capability 相容

5.3 Core Stub Emit

編譯器固定輸出：HCFG → HLAUNCH → HKICK → HWAIT → HPOLL/HERR/HABORT

5.4 Packager

  1. section 對齊與版本資訊
  2. descriptor 壓縮（RLE/delta/formula）
  3. 生成 debug map（wave_id → source）
  4. 支援輸出 HAP container 與 ELF container（payload 語意等價）

6. 封裝格式

6.1 Header

struct HaccPkgHeader {
	uint32_t magic;          // 'HACP'
	uint16_t version_major;
	uint16_t version_minor;
	uint32_t section_count;
	uint32_t entry_offset;   // Native ISA entry
	uint32_t flags;          // bit0: little-endian, bit1: desc-compressed
	uint32_t target_caps;    // capability bitmap
};

6.2 Section Type

enum SectionType : uint32_t {
	SEC_META           = 0,
	SEC_CORE_STUB      = 1,
	SEC_JOB_CONFIG     = 2,
	SEC_WAVE_DESC      = 3,
	SEC_PROFILE_TABLE  = 4,
	SEC_DMA_DESC       = 5,
	SEC_PE_PROGRAM     = 6,
	SEC_SCAN_CHAIN     = 7,
	SEC_DEBUG_MAP      = 8,
	SEC_SYMBOL         = 9
};

6.3 Loader 順序

  1. 驗證 magic/version/target_caps
  2. 載入 PE/scan-chain/profile/dma/wave descriptors
  3. 設定 PC = SEC_CORE_STUB.entry
  4. 啟動 HACC Native Core

6.4 ELF 輸出格式定義（新增）

目的：讓編譯輸出可直接被標準 ELF tooling 識別與載入，同時保留 HACC descriptor-driven 執行模型。

6.4.1 ELF 基本屬性

  1. ELFCLASS32（32-bit）
  2. ELFDATA2LSB（little-endian）
  3. e_type = ET_EXEC（MVP）
  4. e_machine = EM_NONE（MVP；後續可升級為 HACC 專用 machine id）
  5. e_entry 指向 .hacc_core_stub 起始位址

6.4.2 Section 命名與對映

ELF Section對應既有 payload說明
.hacc.metaSEC_META版本/target/caps
.hacc.core_stubSEC_CORE_STUBNative ISA core 程式
.hacc.jobSEC_JOB_CONFIGJob config
.hacc.waveSEC_WAVE_DESCWave descriptors
.hacc.profileSEC_PROFILE_TABLEHDDU profile table
.hacc.dmaSEC_DMA_DESCDMA descriptors
.hacc.peSEC_PE_PROGRAMPE program blob
.hacc.scanSEC_SCAN_CHAINScan chain data
.hacc.debugSEC_DEBUG_MAP除錯映射
.hacc.symSEC_SYMBOL符號資訊（可選）

6.4.3 Program Header（Segment）建議

  1. PT_LOAD RX：.hacc.core_stub
  2. PT_LOAD RO：.hacc.meta/.hacc.job/.hacc.wave/.hacc.profile/.hacc.dma
  3. PT_LOAD RO：.hacc.pe/.hacc.scan
  4. PT_NOTE（可選）：build id / compiler version / target_caps

6.4.4 載入規範（ELF 路徑）

  1. Loader 解析 ELF header 與 program headers。
  2. 依 section 名稱定位 HACC payload，搬運到目標記憶體。
  3. 將 .hacc.job/.hacc.wave base 寫入對應 CSR。
  4. 令 PC = e_entry，啟動 Core。

6.4.5 相容性規則

  1. .hap 與 .elf 的 descriptor 內容必須語意等價。
  2. 相同輸入下，兩格式需產生一致的 LAST_WAVE_ID/ERR_CODE 行為。
  3. 壓縮旗標可透過 ELF note 或 .hacc.meta 同步宣告。

7. Core 程式範本（HACC Native ISA）

# r0 固定 0
# r1: job_cfg_ptr
# r2: wave_begin
# r3: wave_count
# r4: wait_mask
# r5: status
# r6: err

start:
	MOVI   r1, JOBCFG_PTR_LO
	HCFG   r1

	MOVI   r2, 0
	MOVI   r3, WAVE_COUNT
	HLAUNCH r2, r3
	HKICK

wait_done:
	MOVI   r4, 0b011
	HWAIT  r4
	HPOLL  r5

	MOVI   r6, 0x4
	AND    r5, r5, r6
	CMP    r5, r0
	BRZ    finish

on_error:
	HERR   r6
	HABORT
	HBREAK

finish:
	HBREAK

8. 大量 Wave（數千～數萬）策略

8.1 核心策略

  1. Code 固定，資料擴張：code O(1), data O(Nwave)
  2. Wave 以 chunk 提交（建議每批 256~2048）
  3. DFU 預抓下一批 descriptor 降低停頓

8.2 描述壓縮

  • delta profile
  • run-length wave
  • formula wave
addr = base0 + i*delta_i + j*delta_j + k*delta_k

8.3 排程模式

  • in_order：穩定、易除錯
  • overlap_prefetch：預設模式，適度重疊
  • aggressive：高吞吐，需更嚴格 buffer 管理

8.4 錯誤恢復

  1. wave completion bitmap 記錄已完成項目
  2. 可從 wave_begin = last_success + 1 續跑
  3. timeout wave 可標記 bad-wave 並依 policy skip/abort

9. 客製化 Unit 硬體功能與行為

9.1 DFU

抓取 wave/profile/dma 相關 descriptor，供 scheduler FIFO 使用。

錯誤：table 越界、格式非法、memory timeout。

9.2 WSU

依 sync/scheduler policy 進行 dispatch 與 inflight window 管理。

模式：in_order / overlap_prefetch / aggressive。

9.3 PAU

將 profile 套用到 HDDU/AGU，支援 full / delta。

寫入順序：iter → stride → base → mask → ctrl。

9.4 CBU

將 cluster bitmap 展開為有序 MMIO writes。

timeout 對應 ERR_CODE=0x03。

9.5 DRU

掃描 DMA_DESC、配置 channel、追蹤 inflight counter。

錯誤：DMA status / deadlock / 非法 descriptor。

9.6 STU

實作 wait 條件與 watchdog；timeout 時寫回錯誤資訊。

policy=skip 可繼續、policy=abort 則全系統 stop。

9.7 MSU

串行化 MMIO、控制可見性與 flush 規則。

同 target register 寫入不可重排。

9.8 EIU

匯總錯誤、產生 IRQ 與 debug break。

優先級：descriptor 越界 > DMA 致命 > MMIO timeout > policy 非法。

10. 版本化與 capability

10.1 Capability Bitmap

  1. bit0：CBU
  2. bit1：DRU
  3. bit2：WSU overlap
  4. bit3：PAU delta
  5. bit4：completion bitmap
  6. bit5：debug step mode

10.2 編譯器降級策略

  1. 無 bit3：強制 profile full write
  2. 無 bit2：強制 in_order
  3. 無 bit4：不輸出可恢復執行資訊

11. 驗證與可觀測性

11.1 最低 trace 事件

  • EV_WAVE_DISPATCH(wave_id)
  • EV_WAVE_DONE(wave_id)
  • EV_DMA_START(desc_id,ch)
  • EV_DMA_DONE(desc_id,ch)
  • EV_ERROR(code,aux)

11.2 建議覆蓋測試

  1. 單波 smoke（full profile）
  2. 多波 overlap（含 prefetch/drain 重疊）
  3. timeout/error injection
  4. 續跑（resume from last_success）

12. Boot/Load Ownership 與啟動時序

12.1 Ownership

  1. 外部 Loader：解析 .hap 或 .elf、搬運 sections、寫入啟動 CSR。
  2. HACC Native Core：執行 HCFG/HLAUNCH/HKICK/HWAIT，錯誤時 HERR/HABORT。
  3. Execution Complex：讀 descriptor、展開 MMIO/DMA 微流程、回報 done/error。

12.2 載入流程

  1. Loader 檢查 magic/version/target_caps
  2. 搬運 sections 至目標位址
  3. 寫入 HACC_JOB_BASE_LO/HI、HACC_WAVE_BEGIN、HACC_WAVE_COUNT
  4. 設 PC = SEC_CORE_STUB.entry 並釋放 reset
  5. Core 執行：HCFG → HLAUNCH → HKICK → HWAIT
  6. done 回報完成；error 走 HERR → HABORT

12.3 責任邊界（必須遵守）

  • HACC Core 不解析 .hap
  • HACC Core 不解析 .elf
  • HACC Core 不負責 大量 section 搬運
  • DFU 僅讀 descriptor，不承擔 package parser

13. readable program text file 格式（Native ISA）

13.1 檔案結構

program <name> {
	 meta { ... }

13.2 區塊語意

  1. meta：版本/平台/cluster_count/deploy_mask
  2. hardware：stride/base、dma base、硬體限制
  3. grouping：cluster 分群別名

  9. wave_graph：依賴與執行策略
  10. runtime_policy：timeout/retry/error_action
  11. launch_plan：切批策略（可選）

13.3 最小欄位約束

  1. cluster_count/deploy_mask/cluster_stride/dma_base 必填
  2. wave_graph 每筆必含 cluster/profile/program/prefetch/drain/sync
  3. runtime_policy.timeout_cycle 必填

15. readable text file 範例（節錄）

15.1 Conv2D

program conv2d_k3c4 {
	 meta { version="1.0" target="sim" cluster_count=2 deploy_mask=0b0011 }
	 hardware { cluster_stride=0x00010000 dma_base=0x80000000 }
	 grouping { group g_conv clusters=[0,1] }
	 pe_programs {
			pe_program conv_main {
				 format="pe-asm-v1"
				 deploy={group:g_conv, pe_range:"0..47"}
				 source_file="pe/conv_main.peasm"
			}
	 }
	 scan_chain {
			order=reverse
			words=[0x40100208,0x40100208,0x40100208,0x40100208]
	 }
	 hddu_profiles { profile conv_wave { plane_en=0xF plane_mode=0x1 max_outstanding=16 } }
	 dma_plan {
			desc load_w   { ch=0 src="dram://W.bin"  dst=0x00002000 bytes=0x9000 }
			desc load_x0  { ch=1 src="dram://X0.bin" dst=0x00020000 bytes=0x24000 }
			desc store_y0 { ch=2 src=0x00070000 dst="dram://Y0.bin" bytes=0x80000 }
	 }
	 wave_graph {
			wave 0 {
				 cluster_mask=0b0011
				 profile_id=conv_wave
				 pe_program_id=conv_main
				 prefetch=[load_w,load_x0]
				 drain=[store_y0]
				 sync_policy="wait_cluster_then_drain"
			}

17. 總結

本規格將 Core Controller 明確定義為 HACC Native ISA 控制前端， 並將主體複雜度放在可觀測硬體單元（DFU/WSU/PAU/CBU/DRU/STU/MSU/EIU）。

  1. ISA 可獨立實作（含編碼、狀態、例外）
  2. 各客製化單元具清楚功能與狀態行為
  3. 以 descriptor 驅動大規模 wave，兼顧吞吐與可維護性
  4. 編譯輸出可選 .hap 或 .elf，利於標準工具鏈整合
